# Пошагово: что сделать дальше

Чеклист действий после того, как плагин готов. Выполняйте по порядку.

---

## Шаг 1. Финальная проверка плагина

- [ ] **Версия:** в `b24-leads-wp.php` и в `readme.txt` (Stable tag) указана одна и та же версия (например 1.0.0).
- [ ] **Тест:** на тестовом WordPress отправьте заявку с формы (CF7 или другой) и убедитесь, что в Битрикс24 появляется лид/сделка с нужными полями.
- [ ] **ZIP:** соберите архив из корня репозитория (команда в корневом README или в `docs/PUBLISHING.md`, раздел «Сборка ZIP»). Распакуйте и проверьте, что внутри одна папка `b24-leads-wp/` с файлами плагина.

---

## Шаг 2. Репозиторий на GitHub

1. На [github.com](https://github.com) создайте новый репозиторий (например `b24-leads-wp`).
2. **Публичный или приватный:** для плагина, который будет в каталоге WordPress.org, код всё равно станет доступен всем. Публичный репо удобен для истории, форков и ссылки в описании. Приватный — если не хотите дублировать код нигде, кроме WordPress.org. Выберите по желанию.
3. Подключите **корень репозитория** (папка, в которой лежат `plugin/` и `site/`) к репо и сделайте первый коммит:
   ```bash
   cd /path/to/b24-leads-wp
   git init
   git add .
   git commit -m "Initial commit: plugin + site stub, v1.0.0"
   git branch -M main
   git remote add origin https://github.com/ВАШ_ЛОГИН/b24-leads-wp.git
   git push -u origin main
   ```
   В репозитории будут папки `plugin/` (плагин WP) и `site/` (Laravel для подписок). Файлы из `plugin/.gitignore` (CONTEXT_*, BITRIX24_*, docs/LOCAL_TESTING.md, docs/PUBLISHING.md) в репо не попадут.

---

## Шаг 3. Публикация в каталоге WordPress.org

1. **Аккаунт:** зайдите на [wordpress.org](https://wordpress.org), войдите или зарегистрируйтесь.
2. **Подать заявку на плагин:** откройте [wordpress.org/plugins/developers/add/](https://wordpress.org/plugins/developers/add/). Заполните форму (название плагина, описание, при желании ссылка на GitHub). Отправьте.
3. **Дождаться одобрения:** после проверки вам пришлют данные доступа к SVN-репозиторию плагина (например `https://plugins.svn.wordpress.org/b24-leads-wp/`).
4. **Выложить код в SVN:**
   - Установите SVN (если ещё нет).
   - Сделайте checkout:
     ```bash
     svn co https://plugins.svn.wordpress.org/ВАШ-SLUG/ b24-wp-svn
     cd b24-wp-svn
     ```
   - Скопируйте в папку `trunk/` содержимое папки **plugin/** (файлы `b24-leads-wp.php`, папки `includes/`, `docs/`, файлы `readme.txt`, `README.md`, `LICENSE`, `CHANGELOG.md` и т.д.). В `trunk/` не должно быть вложенной папки `b24-leads-wp/` или `plugin/` — только корневые файлы плагина.
   - Добавьте файлы и закоммитьте:
     ```bash
     svn add trunk/*
     svn ci -m "Initial commit, version 1.0.0"
     ```
   - Создайте тег версии:
     ```bash
     svn cp trunk tags/1.0.0
     svn ci -m "Tag 1.0.0"
     ```
5. **Скриншот (по желанию):** в папку `assets/` в SVN положите `screenshot-1.png` (страница настроек плагина, 1280×960 или 1200×900). В `readme.txt` в trunk уже есть блок `== Screenshots ==`.
6. **Ревью:** плагин попадёт в очередь проверки. Замечания придут на email. Исправьте и закоммитьте правки в trunk. После одобрения плагин появится в каталоге.

Подробнее — в `docs/PUBLISHING.md`, раздел «Пошагово: выгрузка в WordPress.org».

---

## Шаг 4. После появления в каталоге WordPress.org

- [ ] На странице плагина в каталоге проверьте описание, установку и скриншот.
- [ ] В заголовке плагина (`b24-leads-wp.php`) и в `readme.txt` укажите **Plugin URI** и **Author URI** — ссылку на ваш сайт, где предлагается Pro / подписка (если сайт уже есть).
- [ ] Если правили readme/URI — закоммитьте изменения в SVN в `trunk` и при необходимости создайте новый тег (например 1.0.1).

---

## Шаг 5. Ваш сайт (Pro / подписка)

- [ ] В этом репозитории сайт для подписок — папка **site/** (Laravel). Разверните его по инструкции в корневом README и в `site/README.md`; рекомендации по домену и серверу — в `docs/HOSTING.md` (в корне репо).
- [ ] Создайте лендинг: что даёт плагин, ссылка на бесплатную версию в WordPress.org, условия Pro (расширенные функции, обновления, поддержка).
- [ ] Настройте приём оплаты (подписка/разовый платёж) и выдачу доступа (личный кабинет, письмо со ссылкой на скачивание Pro и т.п.).
- [ ] В описании плагина на WordPress.org укажите ссылку на этот сайт (Plugin URI), чтобы пользователи знали, где оформить Pro.

---

## Шаг 6. Дальнейшая поддержка

- Обновления бесплатной версии: правки в коде → коммит в SVN в `trunk` → новый тег (например `tags/1.0.1`), в `readme.txt` и в заголовке плагина сменить версию на 1.0.1.
- Обновления Pro (если отдельный пакет): раздаёте через свой сайт после оплаты/подписки.

---

## Краткая последовательность

| № | Действие |
|---|----------|
| 1 | Проверить версию, протестировать отправку в B24, собрать ZIP. |
| 2 | Создать репозиторий на GitHub (публичный или приватный), запушить код. |
| 3 | Подать заявку на добавление плагина на WordPress.org → получить SVN → выложить код в trunk и тег 1.0.0 → пройти ревью. |
| 4 | После публикации: указать Plugin URI на ваш сайт (Pro/подписка). |
| 5 | Запустить сайт с описанием и продажей Pro/подписки. |

Если какой-то шаг нужно расписать подробнее (например, только SVN или только сборка ZIP), откройте `docs/PUBLISHING.md` — там детали по каждому пункту.
